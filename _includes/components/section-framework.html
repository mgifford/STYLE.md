The framework

A style guide is more than a color palette or a PDF that gets ignored after launch. Done well, a style framework defines how a project thinks and communicates — across writing, visual design, code, and AI agent behavior — consistently and in a form every contributor can follow.

STYLES.md draws on established style guide practice — editorial voice guides, design token systems, brand books, and content strategy — and puts the result in a machine-readable file at your repository root where humans, automation, and AI agents can all find and enforce it.

The three pillars

Together, these three open-standard files define not just what you build, but how you build it:

ACCESSIBILITY.md

The foundational requirement. Style is a subset of accessibility. Defines inclusion guardrails and contributor requirements.

SUSTAINABILITY.md

The environmental constraint. Good style is lightweight and efficient. Defines compute and content efficiency rules.

STYLES.md

The expression layer. Voice, tone, design tokens, and AI agent constraints for your specific project.

What a complete style framework covers

Traditional style guides — from editorial manuals to full brand books — address a consistent set of dimensions. STYLES.md covers all of them in a single, portable file:

Voice and tone

How the project sounds — authoritative peer, plain language, active voice, and how tone shifts by context (onboarding, technical, error states).

Grammar and mechanics

Sentence case headings, Oxford comma, active voice, spelling variant, abbreviations, numbers, and date formats. The rules that make every sentence consistent.

Visual language and design tokens

Color, typography, spacing, and breakpoints defined as CSS custom properties so humans and AI agents apply exactly the same values.

Content structure

Heading hierarchy, list types, link text conventions, and how different document types (guides, references, examples) are organized and formatted.

Accessibility requirements

Contrast ratios, alt text, landmark roles, and keyboard behavior — the non-negotiable guardrails that apply to every surface and every contributor.

AI and automation rules

Explicit constraints for AI coding agents: which surface is being edited, which files take precedence, and what must never be overridden automatically.

Style guide, design system, or pattern library?

These terms are often used interchangeably but they describe different tools with different scopes:

  • A style guide defines visual design language: color, typography, spacing, and iconography.
  • A voice and tone guide defines how the project writes, what words it uses, and how it adjusts for context.
  • A pattern library packages reusable UI components and documents their correct usage.
  • A design system combines all of the above — style guide, component library, and governance rules — into one maintained system.

STYLES.md is a style and voice guide with AI enforcement. It pairs with your component library, linter configuration, and CI pipeline rather than replacing them. The three-pillar model extends it to cover accessibility and sustainability as first-class constraints.
